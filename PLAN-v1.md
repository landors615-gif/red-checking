# Red Checking MVP 规划方案 v1

## 1. 项目目标

Red Checking 的 MVP 目标不是做一个“大而全”的小红书分析平台，而是优先跑通一条高价值、可分享、可复用的内容研究闭环：

```text
输入小红书账号/帖子链接
→ 抓取公开内容与基础互动信息
→ AI 输出结构化分析结果
→ 生成可分享的精致报告页
→ 用户浏览 / 转发 / 用于内部汇报
```

这个 MVP 要验证的不是“能抓多少数据”，而是三件事：
1. 用户是否愿意通过输入链接获得分析结果；
2. AI 报告是否有明显价值感；
3. 报告页是否值得分享和传播。

---

## 2. MVP 产品策略收敛

### 2.1 本期只做两类输入
- 小红书账号链接
- 小红书帖子链接

不做：
- 关键词搜索
- 账号名搜索
- 批量导入
- 多账号监控
- 用户中心 / 支付 / 权限系统

### 2.2 本期只做两类分析对象
- **账号检测**：适合竞品研究、账号诊断、汇报场景
- **帖子检测**：适合爆文拆解、单篇内容分析、选题启发

### 2.3 产品核心价值
Red Checking 不是数据看板，而是 **AI 内容研究报告工具**。

它提供的核心价值不是“展示原始数据”，而是：
- 快速整理关键信息
- 结构化输出洞察
- 生成适合汇报和分享的结果页

---

## 3. 推荐的首发定位

### 3.1 推荐首批目标用户
优先建议：
- **医美 / 美妆 / 消费品牌运营团队**
- **代运营团队 / 顾问团队**

原因：
1. 对“小红书竞品/内容研究”需求强；
2. 对“精致报告”价值敏感；
3. 更容易形成复用和转介绍；
4. 更容易接受后续按次收费 / 服务收费。

### 3.2 推荐首发主场景
建议优先级：
1. **账号检测**（主）
2. **帖子检测**（辅）

原因：
- 账号检测更接近老板汇报、客户交付、竞品诊断这类高价值场景；
- 帖子检测更适合作为补充能力，提高产品完整度；
- 如果只先打一个主场景，账号检测更有商业感。

### 3.3 推荐首期验证重点
优先验证：
- **报告有没有价值感**
- **页面有没有传播性**
- **用户是否愿意继续分析第二个链接**

先不把重点放在：
- 复杂商业化
- 复杂长期监测
- 高级权限和后台系统

---

## 4. MVP 功能结构

## 4.1 输入层
### 必须有
- 链接输入框
- 自动识别：账号链接 / 帖子链接
- 提交分析按钮

### 需要处理的异常
- 非小红书链接
- 无法识别的链接格式
- 抓取失败 / 内容不存在 / 登录态失效

---

## 4.2 抓取层

### 账号检测需要的数据
- 账号昵称
- 账号简介
- 主页公开内容列表
- 近期帖子（建议取最近 10-20 条）
- 每条帖子的基础互动数据
- 帖子标题 / 封面 / 发布时间
- 部分热门评论样本（若可抓）

### 帖子检测需要的数据
- 标题
- 正文
- 发布时间
- 标签/话题
- 封面 / 图片信息
- 点赞 / 收藏 / 评论 / 分享等基础互动数据
- 热门评论样本

### 本期原则
- 抓到“足够做出好报告”的数据即可
- 不为了完整而拖慢 MVP

---

## 4.3 AI 分析层

AI 层必须输出 **结构化 JSON**，而不是一段散文。

### 账号检测输出建议结构
- account_summary
- positioning
- target_audience
- content_topics
- viral_posts
- title_patterns
- content_patterns
- comment_insights
- strengths
- weaknesses
- benchmarking
- action_suggestions

### 帖子检测输出建议结构
- post_summary
- theme
- title_analysis
- structure_analysis
- emotional_hooks
- value_points
- conflict_points
- spread_factors
- comment_insights
- benchmark_reference
- improvement_suggestions

### AI 输出原则
- 必须可渲染
- 必须结构统一
- 必须便于页面展示
- 宁可少而稳，不要大而散

---

## 4.4 报告输出层

### 报告页必须有的模块
1. 基础信息卡
2. 账号 / 帖子概览
3. 核心数据摘要
4. AI 核心结论
5. 爆文样本 / 代表内容区
6. 评论区洞察
7. 可执行策略建议
8. 分享入口

### 报告页要求
- 手机端优先
- 页面要有专业报告感
- 支持直接分享链接
- 先做静态可读性，不追求复杂交互

---

## 5. 页面规划

## 5.1 首页
目标：快速让用户理解产品，并完成输入。

### 模块
- 顶部标题与一句话价值主张
- 输入框
- 分析按钮
- 支持的输入类型说明
- 示例报告入口
- 产品卖点说明

### 首页关键原则
- 首页不要太重
- 第一屏就要能输入链接
- 不要做成传统后台首页

---

## 5.2 分析中页面
目标：降低等待焦虑。

### 模块
- 抓取中 / 分析中 / 生成中状态
- 进度提示
- 预计耗时文案
- 失败重试提示

### 关键体验点
- 给用户明确的过程感
- 不要让等待页像“卡死了”

---

## 5.3 报告页
目标：交付结果 + 支持传播。

### 报告页结构建议
1. Hero 概览区
2. 核心判断区
3. 数据摘要区
4. 内容 / 爆文样本区
5. 评论区洞察区
6. 建议动作区
7. 分享/复制链接区

### 设计方向建议
- 不是后台风，而是“咨询报告 + 编辑设计”的混合风格
- 手机优先阅读
- 层级强、信息块清晰
- 适合截图传播和直接转发

---

## 6. 技术架构建议

## 6.1 前端
建议：**Nuxt 3**

原因：
- 适合内容展示型产品
- 适合静态化 / 页面部署
- 后续做分享页和 SEO 也顺手

首期页面：
- `/` 首页
- `/analyzing/:taskId` 分析中页
- `/report/:reportId` 报告页

## 6.2 后端
建议：**FastAPI**

职责：
- 接收链接任务
- 调度抓取
- 统一清洗数据
- 调用 LLM
- 产出结构化 JSON
- 存储报告结果

## 6.3 数据抓取
- 优先复用现有 `xiaohongshu-skills`
- 封装为后端可调用的抓取服务层
- 对抓取异常做标准化处理

## 6.4 报告生成
- 首期直接使用数据库/文件中的结构化 JSON 驱动前端渲染
- 可选：同时生成静态 HTML 快照
- 每份报告拥有独立 URL

---

## 7. 数据流设计

```text
用户提交链接
→ 前端调用 create-analysis-task
→ 后端识别链接类型（账号 / 帖子）
→ 调用抓取服务
→ 原始数据清洗标准化
→ 调用 LLM 输出结构化分析 JSON
→ 保存报告记录
→ 前端轮询任务状态
→ 跳转到报告页
→ 用户浏览 / 转发
```

## 9. 给 COD 的开发拆解

## 9.1 Phase 1：先跑通主链路
### 后端
- 建立 FastAPI 项目骨架
- 提供 `POST /api/analysis` 创建任务
- 提供 `GET /api/analysis/:id` 查询状态
- 提供链接类型识别器
- 打通抓取调用层
- 输出 mock / real 两种分析结果模式

### 前端
- 首页输入与提交
- 分析中页
- 报告页基础骨架
- 状态轮询

### 验收标准
- 输入一个链接后能跑到报告页
- 即使先用 mock 数据，也要完整走通体验

---

## 9.2 Phase 2：接真实分析结果
### 后端
- 建立统一 JSON schema
- 对账号检测与帖子检测分别接入真实 AI 分析
- 增加失败重试与错误状态

### 前端
- 报告页按 schema 渲染完整模块
- 加强手机端体验

### 验收标准
- 同一类报告结构稳定
- 页面可读性达到可演示水平

---

## 9.3 Phase 3：分享与示例报告
### 后端
- 报告生成独立 slug
- 提供公开访问链接

### 前端
- 增加分享区
- 增加示例报告入口

### 验收标准
- 报告可直接发给别人打开
- 首页可展示示例报告

---

## 10. 我对 COD 的具体要求

1. **先跑通，不要一开始过度工程化**
2. **先把 schema 定稳，再做花活**
3. **前端优先手机端**
4. **报告页质量优先于后台能力**
5. **抓取失败、AI 失败、空数据情况必须有兜底状态**
6. **所有模块围绕“可展示结果”服务，不先做后台系统**

---

## 11. 当前建议的设计决策

如果由我先拍板，我建议：

- 首批用户：**医美/美妆/消费品牌团队 + 顾问/代运营团队**
- 首发主场景：**账号检测优先，帖子检测补足**
- 报告主要用途：**给老板/客户看，也兼顾运营研究使用**
- 免费边界：**每日少量免费 + 简版结果**
- MVP 验证重点：**报告价值感 + 分享传播性 + 二次使用意愿**

## 13. 一句话总结

Red Checking 的 MVP 成败，不取决于抓取多复杂，而取决于：

> **能不能把“输入链接 → 输出值得传播的专业报告”这条路径做得顺滑、稳定、好看、有洞察。**
