# 1. 文档说明


## 1.1 文档目的
本文档用于明确 Red Checking 产品在 MVP 阶段的业务目标、用户对象、产品范围、功能设计、技术方案、项目节奏及风险边界，作为后续产品评审、设计、研发与测试协同的统一依据。

## 1.2 产品名称
Red Checking

## 1.3 产品一句话定义
Red Checking 是一款面向品牌方、运营团队、机构与内容从业者的小红书账号/内容检测分析工具。用户输入小红书账号链接或帖子链接后，系统自动抓取公开内容并结合 AI 生成结构化分析报告，帮助用户快速判断账号质量、内容策略、爆文规律和可借鉴方向。

## 1.4 当前阶段定义
当前阶段为 MVP 阶段，核心目标不是做“大而全”的小红书分析平台，而是优先验证以下三件事：
1）用户是否愿意使用链接输入方式获取小红书分析结果；
2）AI 输出的分析报告是否具备明显价值感；
3）可分享的报告页面是否具备传播性与商业转化潜力。


# 2. 产品背景


## 2.1 行业背景
在品牌营销、内容运营、医美/美妆/消费品牌竞争日趋激烈的环境下，小红书已成为品牌种草、内容获客、竞品监测和用户需求洞察的重要阵地。当前大量团队依赖人工方式进行小红书账号研究，主要流程包括：搜索账号、逐篇查看帖子、截屏、手工汇总数据、人工总结内容规律，再整理为 PPT 或文字报告。

## 2.2 现存问题
当前行业通用做法存在如下明显问题：

（1）分析成本高
人工研究一个账号通常耗时 1-3 小时，若涉及评论区深度阅读、竞品对比或内部汇报，耗时更长。

（2）信息高度分散
账号主页、内容列表、互动数据、评论区反馈、内容风格判断等信息分布在不同页面中，不利于快速形成系统结论。

（3）结果不标准化
不同人输出的结论差异较大，同一账号在不同团队手里会被分析成完全不同的结果，缺乏统一框架。

（4）报告难复用
目前大多数账号研究依赖截图+PPT，不仅制作低效，也难以标准化复用，更不利于分享传播。

（5）洞察难转化为动作
即便用户能看出“这个账号做得不错”，也常常无法进一步回答：它为什么有效？哪些内容值得学？哪些打法可以复制？

## 2.3 产品机会
Red Checking 的机会在于，将“小红书账号/内容研究”从一个低效、手工、依赖经验的咨询式动作，升级为一个标准化、可分享、可复用、可扩展的产品能力。


# 3. 产品目标


## 3.1 核心目标
在 MVP 阶段，Red Checking 的目标是跑通以下闭环：

```text
用户输入账号/帖子链接
→ 系统自动抓取内容与基础互动信息
→ AI 输出结构化分析结论
→ 系统生成一份可分享的精致报告页
→ 用户完成浏览、转发或用于内部汇报
```

## 3.2 阶段性验证目标
MVP 重点验证以下问题：

（1）需求成立性
用户是否有高频“小红书账号/帖子分析”需求。

（2）结果价值感
AI 生成的分析结果是否具备“值得看、值得转发、值得保存”的价值。

（3）产品形态可行性
“输入链接 → 输出报告”的产品形态是否比传统人工分析更高效。

（4）商业化基础
该产品后续是否具备 ToB/ToC 的收费空间，如按次收费、订阅制、顾问服务、团队版等。

## 3.3 非目标
以下事项不属于当前 MVP 阶段重点：

1）复杂的多账号长期监控；
2）完整用户中心与支付体系；
3）企业级权限管理；
4）全行业舆情监测；
5）大规模自动日报系统；
6）复杂 BI 数据大屏。


# 4. 目标用户与使用场景


## 4.1 目标用户

### 4.1.1 核心用户群

A. 品牌方 / 市场团队
主要诉求：
- 研究竞品在小红书上的运营策略；
- 判断某个账号是否具备合作价值；
- 快速获得汇报级别的内容分析结果。

B. 医美 / 美妆 / 消费品牌运营团队
主要诉求：
- 分析同行账号打法；
- 研究爆文规律；
- 从评论区中提炼真实用户需求。

C. 代运营团队 / MCN / 顾问公司
主要诉求：
- 快速生成客户用的账号诊断报告；
- 降低人工分析成本；
- 提升方案输出效率和专业感。

D. 创作者 / 博主本人
主要诉求：
- 诊断自身账号内容结构与问题；
- 识别哪些帖子值得复盘；
- 提炼选题与表达策略。

4.1.2 次级用户群
- 投资/咨询团队：用于赛道内容生态判断；
- 电商团队：用于种草内容研究；
- 渠道团队：用于 KOL/KOC 初筛。

### 4.2 典型使用场景

场景一：竞品账号诊断
用户输入某竞品账号链接，希望快速获取该账号定位、内容结构、爆文规律、评论区需求和策略建议。

场景二：单篇爆文检测
用户输入某篇爆文链接，希望知道这篇内容为什么有传播效果、标题和内容结构如何设计、评论区反映了哪些真实需求。

场景三：汇报型报告输出
用户希望得到一份可以直接发给老板、客户、同事的精致分析页面，而不是原始数据或零散截图。

场景四：策略研究与选题启发
用户基于一个账号或一篇帖子，希望获取进一步的内容研究方向和选题启发。

5. 产品定位与价值主张
━━━━━━━━━━━━━━━━━━

5.1 产品定位
Red Checking 不是通用舆情平台，也不是简单的数据面板工具。它的定位是：

“面向小红书内容研究场景的 AI 检测与分析报告工具。”

它强调的是：
- 内容视角，而非纯数据视角；
- 结构化结论，而非原始信息堆叠；
- 报告交付，而非单纯抓取；
- 可分享、可汇报、可复用，而非一次性阅读。

5.2 核心价值主张

对外主张建议：
- 输入一个小红书链接，快速看懂它为什么值得研究。
- 用 AI，一键生成小红书账号/内容检测报告。
- 把竞品分析从手工截图，变成可分享的专业报告。

6. 功能范围定义（MVP）
━━━━━━━━━━━━━━━━━━

6.1 功能总览
MVP 阶段建议以“账号检测 + 帖子检测”为核心，构成基础双场景能力。

产品功能分为四层：
1）输入层；
2）数据抓取层；
3）AI 分析层；
4）报告输出层。

6.2 输入层

6.2.1 必须支持
- 输入小红书账号链接；
- 输入小红书帖子链接。

6.2.2 可选支持（非首发必须）
- 关键词输入；
- 账号名搜索；
- 收藏夹导入；
- 批量链接输入。

6.3 数据抓取层

6.3.1 账号级数据
需支持抓取：
- 账号昵称；
- 账号简介；
- 主页公开内容列表；
- 近期帖子列表；
- 基础互动数据（点赞、收藏、评论、分享）；
- 可见封面与标题信息。

6.3.2 帖子级数据
需支持抓取：
- 标题；
- 正文；
- 发布时间；
- 标签/话题；
- 图片/封面信息；
- 基础互动数据。

6.3.3 评论级数据
需支持抓取：
- 热门评论；
- 高频问题；
- 用户关注点；
- 评论情绪与意图特征（通过 AI 进行二次结构化提取）。

6.4 AI 分析层

6.4.1 账号分析输出
系统需要输出以下结构化结论：
- 账号定位判断；
- 目标受众推测；
- 内容主题分布；
- 爆文样本识别；
- 标题风格总结；
- 内容结构模型总结；
- 评论区需求洞察；
- 账号优势与问题；
- 同类账号的对比；
- 同类账号的数据参考；
- 可借鉴策略建议，账号规划有哪些改进空间。

6.4.2 帖子分析输出
系统需要输出以下结构化结论：
- 内容主题判断；
- 标题拆解；
- 正文结构拆解；
- 情绪点 / 利益点 / 冲突点分析；
- 传播因子判断；
- 评论区用户反馈总结；
- 与同类爆文帖子的数据对比；
- 有哪些改进空间，给出改进建议。

6.5 报告输出层

6.5.1 报告页必要模块
MVP 阶段的报告页至少包含：
1）基础信息卡；
2）账号/帖子概览；
3）关键数据摘要；
4）AI 核心结论；
5）爆文样本区；
6）评论区洞察；
7）策略建议；
8）可分享链接。

6.5.2 报告体验要求
- 支持网页/手机双端浏览；
- 视觉上具有“专业报告感”而非普通后台风格；
- 页面可直接分享；
- 不强求复杂交互，但必须清晰、精致、结构化。

7. 页面结构设计
━━━━━━━━━━━━━━━━━━

7.1 首页
目标：承接用户输入需求，并传达产品价值。

页面功能：
- 输入账号链接 / 帖子链接；
- 简要介绍 Red Checking 的用途；
- 展示示例报告入口；
- 展示核心卖点（如：自动分析 / 评论洞察 / 可分享）。

7.2 分析进行页
目标：降低用户等待焦虑，建立过程可感知性。

页面功能：
- 显示抓取中 / 分析中 / 生成中状态；
- 给予预估等待时长提示；
- 避免页面无反馈造成“失败”误解。

7.3 报告页
目标：完整交付分析结果，并具备分享价值。

页面功能：
- 展示完整分析内容；
- 适合手机端展示；
- 支持复制链接 / 转发；
- 后续可扩展导出功能。

8. 技术方案（MVP）
━━━━━━━━━━━━━━━━━━

8.1 技术架构建议

前端：
- Vue 3 / Nuxt（优先）或 React / Next（二选一）；
- 响应式页面，重点打磨首页、等待页、报告页。

后端：
- Python FastAPI；
- 承担抓取调度、AI 分析调用、结果结构化输出。

数据抓取：
- 复用现有 xiaohongshu-skills 能力；
- 优先支持账号与帖子详情抓取；
- 后续再逐步扩展搜索和批量能力。

AI 分析：
- 使用 LLM 输出结构化内容结论；
- 通过 prompt 模板控制输出框架和稳定性。

报告生成：
- 首期建议生成静态 HTML 报告页；
- 每份报告拥有独立 URL；
- 可部署在 GitHub Pages / OSS / 轻量云环境。

8.2 技术约束与注意事项
- 数据抓取稳定性依赖登录态与平台风控情况；
- 报告生成必须优先保证流程跑通；
- 首期不建议在数据层做过度复杂的深度统计建模；
- 优先把“结果价值感”跑出来，再考虑大规模工程化。

9. 商业模式（草案）
━━━━━━━━━━━━━━━━━━

9.1 免费版
- 每日有限次数；
- 仅提供基础分析结果或简版报告；
- 用于拉新和验证传播链路。

9.2 Pro 版
- 完整报告；
- 更多分析次数；
- 更深的评论区洞察；
- 后续支持导出与对比功能。

9.3 服务版
- 面向品牌、机构、顾问团队；
- 提供定制研究、专项分析、行业监测、策略报告等能力。

10. 核心指标设计
━━━━━━━━━━━━━━━━━━

10.1 产品指标
- 报告生成成功率；
- 报告打开率；
- 报告分享率；
- 报告平均停留时长。

10.2 内容质量指标
- 用户主观反馈中“有帮助”的比例；
- 分析报告是否能带来明确行动建议；
- 用户是否愿意继续分析第二个账号或第二篇帖子。

10.3 商业化指标（后续）
- 免费转付费率；
- 单次报告购买率；
- 团队/顾问服务转化率。


# 11. 风险与边界


11.1 数据抓取风险
- 平台反爬与风控；
- 登录态失效；
- 数据字段获取不完整；
- 评论抓取深度波动。

11.2 AI 输出风险
- 输出结论“看起来合理但不具备实用价值”；
- 观点泛化、缺乏行业语境；
- 输出不稳定，造成报告质量波动。

11.3 产品体验风险
- 生成时间过长；
- 页面过于模板化；
- 用户认为“好看但无用”。

11.4 商业与合规风险
- 数据使用边界；
- 商业化后对平台规则的更高敏感性；
- 报告中引用内容的展示尺度需谨慎控制。

12. 研发实施里程碑
━━━━━━━━━━━━━━━━━━

Phase 1：流程跑通
- 输入链接；
- 成功抓取内容；
- 输出结构化 JSON 分析结果。

Phase 2：页面完成
- 首页完成；
- 等待页完成；
- 报告页完成；
- 手机端可稳定浏览。

Phase 3：分享可用
- 每份报告拥有独立 URL；
- 可直接转发；
- 沉淀 1-3 份示例报告。

Phase 4：小范围试用
- 邀请真实用户试用；
- 收集反馈；
- 优化 prompt 结构与报告排版。

14. 当前建议的产品决策问题
━━━━━━━━━━━━━━━━━━

在进入设计和开发前，建议优先明确以下决策点：

1）首批目标用户是谁？
是先打医美/美妆/消费品牌团队，还是先服务代运营/顾问团队？

2）首发主场景是什么？
是“账号检测”为主，还是“帖子检测”为主？

3）报告的主要用途是什么？
是给老板看、给客户看，还是给运营自己研究使用？

4）免费与付费边界如何划分？
哪些能力免费开放，哪些能力作为 Pro 或服务型交付？

5）首期验证重点是什么？
验证传播性、验证复购、验证付费，还是验证交付效率？

━━━━━━━━━━━━━━━━━━
15. 结论
━━━━━━━━━━━━━━━━━━

Red Checking 的 MVP 方向具备明确的问题基础和现实使用场景。其最关键的成功条件不在于“抓得多复杂”，而在于是否能做到以下三点：

1）分析流程足够顺滑；
2）报告内容足够有洞察；
3）页面结果足够值得分享。

如果能够率先把“链接输入 → 自动分析 → 精致报告”这条路径做扎实，Red Checking 有机会成为小红书内容研究场景中的标准化工具能力，并向更强的监测、对比、SaaS 化方向扩展。
